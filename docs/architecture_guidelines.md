# Архитектурные стандарты и связность ресурсов (Topology Guidelines)

Данный документ описывает правила логической связности инфраструктуры. Эти правила гарантируют, что сгенерированный код будет не просто набором разрозненных ресурсов, а работоспособной топологией.

## 1. Иерархия сетевых ресурсов (VPC)

Правило: Ни одна виртуальная машина или база данных не может существовать вне подсети.

- Условие: Если пользователь просит создать виртуальную машину или БД, но не передает ID уже существующей сети, ты ОБЯЗАТЕЛЬНО должен сгенерировать базовую сеть и подсеть.
- Связность: Все созданные вычислительные ресурсы должны ссылаться на эту подсеть.

## 2. Консистентность зон доступности (Availability Zones)

Правило: Ресурсы должны физически находиться в одной зоне доступности.

- Условие: При создании виртуальных машин и подсетей параметр зоны должен строго совпадать. Выбирай одну зону для всех связанных ресурсов по умолчанию.

## 3. Вывод критически важных данных (Outputs)

Правило: Сгенерированная инфраструктура не должна быть "черным ящиком".

- Условие: Если в результате запроса создаются виртуальные машины, балансировщики или базы данных, ты ОБЯЗАТЕЛЬНО должен добавить блок вывода в конец файла, который покажет их IP-адреса.
- Обоснование: Это необходимо, чтобы другие инструменты могли подключиться к этим адресам для настройки.

## 4. Универсальность (Не плоди лишнее)

- Если пользователь просит создать _только сеть_ или _только Security Group_, НЕ генерируй виртуальные машины.
- Выполняй строго то, что требует пользователь, но обеспечивая полную внутреннюю связность того, что ты сгенерировал.

## 5. Специфика облачных провайдеров (Provider-Specific Logic)

Правило: Обеспечивай полную логическую работоспособность инфраструктуры с учетом архитектуры выбранного провайдера.

- Условие: Если для обеспечения сетевой связности или доступа в интернет требуются специфичные ресурсы целевого провайдера, ты ОБЯЗАН добавить и связать их самостоятельно. Опирайся на свои внутренние знания синтаксиса провайдера, даже если эти ресурсы прямо не указаны в базовых корпоративных правилах.

## 6. Правило изоляции контуров (Secure by Default)

При проектировании сетевой инфраструктуры строго разделяй публичный и приватный доступ:

- **Приватный контур (По умолчанию):** Если нет явного запроса на доступ извне, инфраструктура изолируется. Запрещены публичные IP и маршруты в интернет.
- **Публичный контур:** При требовании публичного сервера ты ОБЯЗАН явно сгенерировать полноценную топологию доступа: шлюз (Gateway) и таблицу маршрутизации (Route Table) с выходом в `0.0.0.0/0`. Категорически запрещено использовать только неявные флаги-ярлыки (например, просто `nat = true` без создания самого шлюза), если провайдер поддерживает явное создание сетевых компонентов.

## 7. Строгий порядок атомарности доступа (Access Guarantee)

Любой код, управляющий пользователями или SSH (Ansible tasks, Terraform cloud-init/runcmd), должен защищать сервер от блокировки (lockout). Ты ОБЯЗАН соблюдать следующую строгую последовательность выполнения сверху вниз:

1. **Создание:** Сначала создается пользователь (например, `corp-admin`).
2. **Ключ:** Затем прописывается его публичный SSH-ключ.
3. **Ужесточение (ОБЯЗАТЕЛЬНО):** Только ПОСЛЕ добавления ключа ты ОБЯЗАН в этом же скрипте отключить пароли. Добавь команды или задачи для установки `PasswordAuthentication no` и `PermitRootLogin no` в `/etc/ssh/sshd_config` и перезапусти службу SSH. Пропуск этого шага недопустим.

## 8. Политика абсолютной изоляции контента (No External Files)

Сгенерированный сценарий должен быть самодостаточным и готовым к запуску «из коробки» на любой пустой машине.

- **Запрет локальных файлов:** КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО использовать любые функции или плагины для чтения локальных файлов с диска пользователя.
- **Триггеры нарушений:** НИКОГДА не используй конструкции `${file("...")}` в Terraform или `lookup('file', '...')` в Ansible.
- **Решение:** Для всех конфигураций и SSH-ключей генерируй текстовые заглушки прямо в коде (inline string). Например, для SSH-ключа всегда пиши строку напрямую: `"ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQ... dummy-key"`.
